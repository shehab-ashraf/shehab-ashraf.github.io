## 1. Initial Baseline Setup

Part of the goal of this project is to **learn as I go**, so I'm starting at the very beginning with simple implementation.

### 1.1 Model Architecture

**NanoGPT** with the following configuration:

```python
class GPTConfig:
    vocab_size: int = 50257      # vocab size
    context_length: int = 1024   # Maximum sequence length
    n_embed: int = 768           # Embedding dimension
    num_layers: int = 12         # Number of transformer blocks
    n_head: int = 12             # Number of attention heads
```

The model uses a standard transformer decoder architecture with:
- **Token + Position Embeddings**: Learned embeddings for both tokens and positions
- **12 Transformer Blocks**: Each containing multi-head attention and MLP
- **Pre-norm Architecture**: LayerNorm before attention and MLP (better gradient flow)
- **Residual Connections**: Clean residual stream through all layers
- **GELU Activation**: Standard GPT-2 activation

```python
class GPT(nn.Module):
    def __init__(self, config):
        super().__init__()
        self.config = config

        self.transformer = nn.ModuleDict(dict(
            wte = nn.Embedding(config.vocab_size, config.embd_size), # Token embeddings
            wpe = nn.Embedding(config.context_length, config.embd_size), # Position embeddings
            h = nn.ModuleList([Block(config) for _ in range(config.num_layers)]), # Transformer blocks
            ln_f = nn.LayerNorm(config.embd_size) # Final layer norm before output
        ))
        # Language modeling head: maps embeddings back to vocabulary logits
        self.lm_head = nn.Linear(config.embd_size, config.vocab_size, bias=False)
        # weight sharing scheme (reduces 768*50267=~40M params, fewer params, more efficient)
        self.transformer.wte.weight = self.lm_head.weight
```

### 1.2 Weight Initialization

```python
def _init_weights(self, module):
    if isinstance(module, nn.Linear):
        torch.nn.init.normal_(module.weight, mean=0.0, std=0.02)
        if module.bias is not None:
            torch.nn.init.zeros_(module.bias)
    if isinstance(module, nn.Embedding):
        torch.nn.init.normal_(module.weight, mean=0.0, std=0.02)
```

**Why std=0.02?**
- Consistent with Xavier initialization: <span class="text-yellow">1/√(fan_in)</span>
- For 768-dimensional embeddings: <span class="text-yellow">1/√768 ≈ 0.036</span>

**Residual Projection Scaling:**
```python
attn_scale = 1 / (2 * config.num_layers) ** 0.5
```
In a residual network, every block contributes some amount that gets **added** to the residual stream. Each addition increases the variance of the activations. After many layers, this variance accumulation causes the activations to grow too large, making training unstable.We apply a scaling factor  <span class="text-yellow">1/√(2 * num_layers)</span> to the attention branch to control this variance growth and keep activations stable throughout the network. Found in [modded-nanogpt](https://github.com/KellerJordan/modded-nanogpt/blob/master/records/track_1_short/2024-10-10_Muon/eb5659d0-fb6a-49e5-a311-f1f89412f726.txt)

## 2. Training Optimizations

### 2.1 Mixed Precision 
Mixed precision training runs most computations in lower-precision formats (bfloat16 or FP16) to leverage GPU **tensor cores** for higher throughput and roughly half the memory usage.

<img src="{{ site.baseurl }}/assets/img/precision.png" 
     alt="Precision comparison" 
     class="img-wide"
     style="width: 35%; height: auto;">

#### 2.1.1 TensorFloat-32 (TF32)
TF32 speeds up matrix multiplications by using Tensor Cores. The cores take FP32 inputs, reduce the mantissa precision to 10 bits while keeping the full FP32 range through the 8-bit exponent, perform the operations much faster, and return the results in FP32. We can use it by adding one line.
```python
torch.set_float32_matmul_precision('high')
```


#### 2.1.2 True Mixed Precision: FP16/BF16
For bigger speedups and memory cuts, we move the heavy math into 16-bit. That halves memory and unlocks larger batches.

How it works: PyTorch <span class="text-yellow">torch.autocast</span> auto-selects precision. In the forward pass, big ops (matrix multiplies) run in 16-bit on tensor cores but sensitive ops (softmax, loss) stay FP32.


<span class="text-yellow">Float16</span> is widely supported and fast, so it works even on older GPUs. Its downside is the limited numeric range, which means small gradients can underflow to zero and stop learning. To avoid this, we use a GradScaler, which scales the loss before backpropagation and unscales the gradients before the optimizer step.

Float16 Recipe:
```python
scaler = torch.cuda.amp.GradScaler() 
with torch.autocast(device_type='cuda', dtype=torch.float16):
    _, loss = model(x, y)
    loss = loss / grad_accum_steps

scaler.scale(loss).backward() # Scale gradients
scaler.step(optimizer)        # Unscale & Step
scaler.update()               # Update scale factor
```

<span class="text-yellow">Bfloat16</span> keeps the same numeric range as FP32 but with fewer mantissa bits. This lets it handle very small gradients and very large activations without the underflow problems of fp16. Because of that, you don’t need gradient scaling. It requires newer GPUs (Ampere and later).


Bfloat16 Recipe:
```python
with torch.autocast(device_type='cuda', dtype=torch.bfloat16):
    _, loss = model(x, y)
    loss = loss / grad_accum_steps
    loss_accum += loss.detach()

# Backward pass (Simple! No scaler needed)
loss.backward()
```

For this project, I am using Bfloat16 Mixed Precision combined with TF32 for maximum throughput.

### 2.2 torch.compile()

One of the most powerful optimizations in PyTorch 2.0 is the magic line:
```python
model = torch.compile(model)
```

In standard "eager" mode, PyTorch executes operations one at a time, launching separate GPU kernels for each operation. Each kernel:
1. Loads data from GPU memory (HBM)
2. Computes the result
3. Writes the result back to HBM

`torch.compile()` uses two key optimizations:

**1. Kernel Fusion**
Combines multiple operations into a single GPU kernel. Instead of writing intermediate results back to HBM between each operation, it keeps them in SRAM.


**2. Graph Optimization**
Analyzes the entire computation graph and:
- Removes redundant operations
- Reorders operations for better memory access patterns
- Selects the fastest CUDA kernels for your specific hardware
- Optimizes memory layout

**Compilation time** - The first time you run the model, PyTorch pauses to analyze your code and generate optimized kernels. This can take 1-2 minutes. But once it's done, every subsequent step is much faster.

### 2.3 Flash Attention

The math is cheap, just two matrix multiplies (Q×K^T and then ×V) plus a softmax.But for 1024 tokens you create a 1024×1024 attention matrix Every entry has to travel from slow **HBM** (the big DRAM on your GPU) to the cores and back.

During the backward pass you need those intermediates again:

- **Store the big matrix**: the 1024×1024 softmax probabilities sit in HBM the whole time.
- **Don't store it, recompute during backprop**: you re-run Q×K^T + softmax from scratch. But every recompute still loads Q and K from slow HBM, writes the huge matrix back to HBM temporarily, then reads it again for the gradient step.

Result: your tensor cores sit idle waiting for data. The bottleneck is **memory bandwidth**, not FLOPs.

Flash Attention fixes exactly that.<span class="text-yellow">**The trick: stay in SRAM.**</span><br>
It tiles Q, K, V into tiny blocks that fit completely inside on-chip SRAM. Then it fuses everything into a single kernel:<br>
1. Load a block of Q and K from HBM to SRAM  
2. Compute Q×K^T  
3. Do softmax *on the fly* using online normalization
4. Multiply by the V block and accumulate  
5. Write only the final output back to HBM  

That's it. No giant attention matrix ever lives in slow memory.

**Recomputation during backprop:** Instead of storing the massive attention matrices (which would require O(N²) memory), Flash Attention recomputes them tile-by-tile in SRAM during the backward pass. Since Q and K must be read from HBM anyway to compute gradients, the extra FLOPs to recompute attention scores are essentially free compared to the cost of storing and reading back the full N×N attention matrix from HBM.

In PyTorch ≥ 2.0 it's literally one line:
```python
attn_output = F.scaled_dot_product_attention(q, k, v, attn_mask=None, dropout_p=0.0, is_causal=True)
```
After training optimizations, the model achieved <span class="text-yellow">200K tokens/sec</span> with a final loss of 7.03 after training on 20M tokens in <span class="text-yellow">2.66 minutes</span> on a single A100 GPU.

## 3. Architectural Changes

There are some basic architectural changes and modernizations that can be made to the model that will speed up training.

### 3.1 Rotary Positional Embeddings (RoPE)

So far, the model encodes position with a learned lookup table:

```python
wpe = nn.Embedding(config.context_length, config.embd_size)  # (1024, 768)
```

One 768-dim vector per position, added to the token embedding before the first transformer block. Clean, works fine during training. But it has two structural problems that limit what the model can learn.


#### Problem 1: No generalization across positions

Each row in `wpe` is trained independently. When the model sees tokens at positions (5, 8) and learns something about their relationship, that knowledge lives in the weights of *W_q* and *W_k* as a response to those specific position embedding values. At positions (505, 508), the position embeddings are completely different vectors. Same structural relationship, but the model has to relearn it from scratch. The model is learning to recognize absolute position IDs, not distances.

#### Problem 2: Fixed context-length
Position 1024 doesn't exist in the table. Any sequence longer than `context_length` is undefined. 

#### What the attention mechanism actually needs

Look at the attention score between query at position *m* and key at position *n*:

$$\text{score}(m, n) = \mathbf{q}_m \cdot \mathbf{k}_n$$

The score should depend on the token content and on the relative offset *(m - n)*. Not on *m* and *n* independently. Tokens at (5, 8) and (505, 508) share the same relative offset of 3, so they should produce the same attention patterns given the same content. What we want is a function *f* applied to queries and keys such that:

$$f(\mathbf{q}, m) \cdot f(\mathbf{k}, n) = g(\mathbf{q}, \mathbf{k}, m - n)$$

#### RoPE: position as rotation

The idea: instead of adding a position vector to the embedding, **rotate the query and key vectors by an angle proportional to their position**.

Start with the simplest case: a 2D vector $$(x_1, x_2)$$ at position $$m$$:

$$R(m\theta) \begin{bmatrix} x_1 \\ x_2 \end{bmatrix} = \begin{bmatrix} \cos(m\theta) & -\sin(m\theta) \\ \sin(m\theta) & \cos(m\theta) \end{bmatrix} \begin{bmatrix} x_1 \\ x_2 \end{bmatrix}$$

Query at position $$m$$, key at position $$n$$. Now compute their dot product:

$$\big(R(m\theta)\,\mathbf{q}\big)^\top \big(R(n\theta)\,\mathbf{k}\big) = \mathbf{q}^\top R\big((n-m)\theta\big)\,\mathbf{k}$$

The $$m$$ and $$n$$ canceled, only the difference $$(n - m)$$ survives. This is exactly the property we wanted, and it comes directly from how rotation matrices compose: $$R(\alpha)^\top R(\beta) = R(\beta - \alpha)$$.

Now scale this up. Each attention head operates on a vector of `head_dim = n_embed // n_head = 768 // 12 = 64` dimensions and the sequence has `context_length = 1024` positions, so RoPE must encode every position from 0 to 1023 into these 64-dimensional vectors. A proper 64D rotation would be a full 64×64 matrix, mixing any dimension with any other. We don't build that. Instead, split the 64 dims into 32 consecutive pairs and apply an independent 2D rotation to each pair:

$$\mathbf{q} = \underbrace{(q_1, q_2)}_{\text{pair 0}},\; \underbrace{(q_3, q_4)}_{\text{pair 1}},\; \ldots,\; \underbrace{(q_{63}, q_{64})}_{\text{pair 31}}$$

This is **not** the same as a full 64D rotation on the vector itself. The two operations produce different vectors. But we don't care about the vector. We care about the dot product $$\mathbf{q} \cdot \mathbf{k}$$. And the dot product decomposes across pairs:

$$\mathbf{q} \cdot \mathbf{k} = \sum_{i=0}^{31} \mathbf{q}_{2i:2i+2} \cdot \mathbf{k}_{2i:2i+2}$$

Rotating pair $$i$$ of $$\mathbf{q}$$ and $$\mathbf{k}$$ only touches term $$i$$ in that sum and has zero effect on every other term. Each term is exactly the 2D case we already proved: it only depends on $$(n - m)$$. This means the full rotation is 32 independent 2×2 rotations:

$$R_{\text{full}}(m) = \text{diag}\Big(R(m\theta_0),\; R(m\theta_1),\; \ldots,\; R(m\theta_{31})\Big)$$

Which in code is just:

```python
def apply_rotary_emb(q: torch.Tensor, k: torch.Tensor, cos: torch.Tensor, sin: torch.Tensor):
    def rotate_half(x):
        x1 = x[..., : x.shape[-1] // 2]
        x2 = x[..., x.shape[-1] // 2 :]
        return torch.cat((-x2, x1), dim=-1)

    q_embed = (q * cos) + (rotate_half(q) * sin)
    k_embed = (k * cos) + (rotate_half(k) * sin)
    return q_embed, k_embed
```

**One question remains:** each pair gets its own frequency $$\theta_i$$. Should they all be the same?


#### Multiple frequencies: solving aliasing

With a single $$\theta$$ for all 32 pairs, positions 0 and $$2\pi$$ are indistinguishable:

$$\cos(0 \cdot 1.0) = \cos(2\pi \cdot 1.0) = 1.0$$

The model sees the same rotational fingerprint at position 0 and position $$2\pi$$, so it can't tell them apart. This is **aliasing**.

The fix: give each pair a different frequency $$\theta_i$$, chosen so that no two positions produce identical fingerprints across *all* 32 pairs simultaneously:

$$\theta_i = \frac{1}{10000^{2i/d}}, \quad i = 0, 1, \ldots, \frac{d}{2}-1$$

Two positions might alias on pair 0 (fast frequency wraps). They won't alias on pair 31 (slow frequency hasn't completed one cycle yet).

<span class="text-yellow">Why base = 10000?</span> The base controls how fast the slowest frequency rotates. For a 64‑dim head, the slowest frequency (i = 31) has a period of about:

$$2\pi \times 10000^{(d-2)/d} \approx 63{,}000 \text{ tokens}$$

That means with base=10000, any two positions less than ~63,000 steps apart have a unique fingerprint across all 32 pairs.  
We train on sequences up to 1024 tokens, so all training positions are far inside that range, hence no aliasing during training.

**The implementation:**

```python
class Rotary(nn.Module):

    def __init__(self, head_dim: int, max_seq_len: int = 1024, base: float = 10000.0):
        super().__init__()
        inv_freq = 1.0 / (base ** (torch.arange(0, head_dim, 2).float() / head_dim))
        self.register_buffer("inv_freq", inv_freq)
        self.max_seq_len = max_seq_len

    def forward(self, seq_len: int, device: torch.device):
        t = torch.arange(seq_len, device=device, dtype=self.inv_freq.dtype)
        freqs = torch.outer(t, self.inv_freq)
        freqs = torch.cat((freqs, freqs), dim=-1)
        cos = freqs.cos()[None, None, :, :]
        sin = freqs.sin()[None, None, :, :]
        return cos, sin
```
### 3.2 RMSNorm
I replaced standard LayerNorm with RMSNorm. LayerNorm centers and scales the input. RMSNorm just scales it. It drops the mean-centering operation and the bias terms, which simplifies the computation and reduces parameters. It preserves model performance while being computationally cheaper and faster.

### 3.3 Squared ReLU Activation
I swapped out the standard GELU activation for ReLU² (Squared ReLU). GELU uses slow math operations like `tanh` and exponentials. ReLU² is just `max(0, x)^2`. Dead simple and fast. It runs better on hardware and often beats GELU in modern LLM training. You get the non-linearity you need without the extra cost.

### 3.4 Vocabulary Padding
The default GPT-2 vocabulary size is *50,257*. This is a nightmare for GPUs, which love powers of 2. I padded the vocabulary size to *50,304* (which is `50257 + 47`). This small change ensures that all tensor dimensions are nice round numbers, preventing memory misalignment and wasted cycles. It’s a tiny change that keeps the Tensor Cores happy.

After these changes, the model achieved <span class="text-yellow">205,000 tokens/sec</span> with a final loss of 6.93 after training on 20M tokens in <span class="text-yellow">1.82 minutes</span> on a single A100 GPU.
