## 1. Initial Baseline Setup

Part of the goal of this project is to **learn as I go**, so I'm starting at the very beginning with simple implementation.

### 1.1 Model Architecture

**NanoGPT** with the following configuration:

```python
class GPTConfig:
    vocab_size: int = 50257      # vocab size
    context_length: int = 1024   # Maximum sequence length
    n_embed: int = 768           # Embedding dimension
    num_layers: int = 12         # Number of transformer blocks
    n_head: int = 12             # Number of attention heads
```

The model uses a standard transformer decoder architecture with:
- **Token + Position Embeddings**: Learned embeddings for both tokens and positions
- **12 Transformer Blocks**: Each containing multi-head attention and MLP
- **Pre-norm Architecture**: LayerNorm before attention and MLP (better gradient flow)
- **Residual Connections**: Clean residual stream through all layers
- **GELU Activation**: Standard GPT-2 activation

```python
class GPT(nn.Module):
    def __init__(self, config):
        super().__init__()
        self.config = config

        self.transformer = nn.ModuleDict(dict(
            wte = nn.Embedding(config.vocab_size, config.embd_size),    # Token embeddings
            wpe = nn.Embedding(config.context_length, config.embd_size),  # Position embeddings
            h = nn.ModuleList([Block(config) for _ in range(config.num_layers)]),  # Transformer blocks
            ln_f = nn.LayerNorm(config.embd_size)  # Final layer norm before output
        ))
        # Language modeling head: maps embeddings back to vocabulary logits
        self.lm_head = nn.Linear(config.embd_size, config.vocab_size, bias=False)
        # weight sharing scheme (reduces 768*50267=~40M params, fewer params, more efficient)
        self.transformer.wte.weight = self.lm_head.weight
```

### 1.2 Weight Initialization

```python
def _init_weights(self, module):
    if isinstance(module, nn.Linear):
        torch.nn.init.normal_(module.weight, mean=0.0, std=0.02)
        if module.bias is not None:
            torch.nn.init.zeros_(module.bias)
    if isinstance(module, nn.Embedding):
        torch.nn.init.normal_(module.weight, mean=0.0, std=0.02)
```

**Why std=0.02?**
- Consistent with Xavier initialization: `1/√(fan_in)`
- For 768-dimensional embeddings: `1/√768 ≈ 0.036`

**Residual Projection Scaling:**
```python
attn_scale = 1 / (2 * config.num_layers) ** 0.5
```
In a residual network, every block contributes some amount that gets **added** to the residual stream. Each addition increases the variance of the activations. After many layers, this variance accumulation causes the activations to grow too large, making training unstable.We apply a scaling factor `1 / (2 * config.num_layers) ** 0.5` to the attention branch to control this variance growth and keep activations stable throughout the network.
Found in [modded-nanogpt](https://github.com/KellerJordan/modded-nanogpt/blob/master/records/track_1_short/2024-10-10_Muon/eb5659d0-fb6a-49e5-a311-f1f89412f726.txt)

### 1.4 Training Configuration

**Hyperparameters:**
```python
total_batch_size = 524288      # 2^19 tokens per optimizer step

```

**Training Loop Structure:**
```python
for step in range(max_steps):
    optimizer.zero_grad()
    loss_accum = 0.0
    # Gradient accumulation
    for micro_step in range(grad_accum_steps):
        x, y = train_loader.next_batch()
        x, y = x.to(device), y.to(device)   
        _, loss = model(x, y)
        loss = loss / grad_accum_steps 
        loss_accum += loss.detach()
        loss.backward()
    optimizer.step()
```
The baseline achieved **16,700 tokens/sec** with a final loss of 7.04 after training on 20M tokens in **19.9 minutes** on a single A100 GPU.


## 2. Training Optimizations

### 2.1 Mixed Precision 
Mixed precision training runs most computations in lower-precision formats (bfloat16 or FP16) to leverage GPU **tensor cores** for higher throughput and roughly half the memory usage.

  </figcaption>

#### 2.1.1 TensorFloat-32 (TF32)
TF32 speeds up matrix multiplications by using Tensor Cores. The cores take FP32 inputs, reduce the mantissa precision to 10 bits while keeping the full FP32 range through the 8-bit exponent, perform the operations much faster, and return the results in FP32. We can use it by adding one line.
```python
torch.set_float32_matmul_precision('high')
```


#### 2.1.2 True Mixed Precision: FP16/BF16
For bigger speedups and memory cuts, we move the heavy math into 16-bit. That halves memory and unlocks larger batches.

How it works: PyTorch `torch.autocast` auto-selects precision. In the forward pass, big ops (matrix multiplies) run in 16-bit on tensor cores but sensitive ops (softmax, loss) stay FP32.

##### Float16 (fp16)
Float16 is widely supported and fast, so it works even on older GPUs. Its downside is the limited numeric range, which means small gradients can underflow to zero and stop learning. To avoid this, we use a `GradScaler`, which scales the loss before backpropagation and unscales the gradients before the optimizer step.

##### Bfloat16 (bf16)
Bfloat16 keeps the same numeric range as FP32 but with fewer mantissa bits. This lets it handle very small gradients and very large activations without the underflow problems of fp16. Because of that, you don’t need gradient scaling. It requires newer GPUs (Ampere and later).

##### Implementation

Bfloat16 Recipe:
```python
with torch.autocast(device_type='cuda', dtype=torch.bfloat16):
    _, loss = model(x, y)
    loss = loss / grad_accum_steps
    loss_accum += loss.detach()

# Backward pass (Simple! No scaler needed)
loss.backward()
```

Float16 Recipe:
```python
scaler = torch.cuda.amp.GradScaler() 
with torch.autocast(device_type='cuda', dtype=torch.float16):
    _, loss = model(x, y)
    loss = loss / grad_accum_steps

scaler.scale(loss).backward() # Scale gradients
scaler.step(optimizer)        # Unscale & Step
scaler.update()               # Update scale factor
```

For this project, I am using **Bfloat16 Mixed Precision** combined with **TF32** for maximum throughput.

### 2.2 torch.compile()

One of the most powerful optimizations in PyTorch 2.0 is the magic line:
```python
model = torch.compile(model)
```

In standard "eager" mode, PyTorch executes operations one at a time, launching separate GPU kernels for each operation. Each kernel:
1. Loads data from GPU memory (HBM)
2. Computes the result
3. Writes the result back to HBM

`torch.compile()` uses two key optimizations:

**1. Kernel Fusion**
Combines multiple operations into a single GPU kernel. Instead of writing intermediate results back to HBM between each operation, it keeps them in SRAM.


**2. Graph Optimization**
Analyzes the entire computation graph and:
- Removes redundant operations
- Reorders operations for better memory access patterns
- Selects the fastest CUDA kernels for your specific hardware
- Optimizes memory layout

**Compilation time** - The first time you run the model, PyTorch pauses to analyze your code and generate optimized kernels. This can take 1-2 minutes. But once it's done, every subsequent step is much faster.

### 2.3 Flash Attention

Standard attention is slow because it hits the **Memory Wall**. For 1024 tokens, it computes a 1024 × 1024 matrix of scores, applies softmax, and multiplies by V. Each step has to read from and write to slow GPU memory (HBM). During the backward pass, these matrices are needed again, doubling the memory traffic. The problem isn’t the math, it’s all the back-and-forth to slow memory.


Flash Attention solves this through **kernel fusion** and **tiling**:

- **Stay Local:** It partitions the attention computation into small tiles that fit entirely in **SRAM**.
- **Fused Computation:** For each tile, Flash Attention:
    1. Loads small blocks of Q, K, and V from HBM into SRAM
    2. Computes attention scores (Q x K^T)
    3. Applies softmax incrementally using online normalization
    4. Multiplies by V and accumulates the result
- **Recomputation Strategy:** During backpropagation, Flash Attention recomputes attention scores on-the-fly from Q, K, V rather than storing them.
